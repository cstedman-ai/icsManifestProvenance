# 20260324 — Mobile Receiving — Scan and Verify

> **Purpose:** Define how the application behaves when accessed from a mobile device browser, restricting to Coreweave-only login for warehouse scanning and receiving. Also detail a Capacitor-based native app option for iOS and Android.

## 1. Rationale

Warehouse staff need to scan QR codes on pallets and verify line items from a phone or tablet at the dock. Vendors, on the other hand, create multi-step shipment manifests — a workflow designed for desktop. Allowing vendor login on mobile creates UX problems (complex wizard on a small screen) and security exposure (vendor credentials on shared warehouse devices).

**Rule:** Mobile devices see only the Coreweave login and are routed directly into a scan-optimized receiving flow.

## 2. Mobile Detection

### 2.1 Detection Method

Create a utility `src/utils/device.js` that exposes a reactive `isMobile` check:

```js
export function isMobileDevice() {
  if (typeof navigator === 'undefined') return false;

  // Capacitor native app — always treat as mobile
  if (window.Capacitor?.isNativePlatform()) return true;

  // Touch + small viewport (covers phones and most tablets in portrait)
  const hasTouch = 'ontouchstart' in window || navigator.maxTouchPoints > 0;
  const narrowViewport = window.matchMedia('(max-width: 768px)').matches;

  // User-agent fallback for edge cases
  const mobileUA = /Android|iPhone|iPad|iPod|webOS|BlackBerry|Opera Mini|IEMobile/i
    .test(navigator.userAgent);

  return (hasTouch && narrowViewport) || mobileUA;
}
```

### 2.2 Why Not User-Agent Alone?

User-agent strings are unreliable and increasingly spoofed. The primary signal is **touch capability + viewport width**, with UA as a fallback. This avoids false positives from desktop touch screens while catching phones and tablets.

### 2.3 Re-Evaluation on Resize

If a user rotates a tablet or connects a keyboard, the app should re-evaluate. Expose a `useMobileDetect` React hook that listens to the `resize` and `orientationchange` events and returns a live `isMobile` boolean.

```
src/utils/device.js
  ├── isMobileDevice()      — one-shot check
  └── useMobileDetect()     — React hook, updates on resize/orientation
```

## 3. Login Behavior on Mobile

### 3.1 What Changes

| Element | Desktop | Mobile |
|---|---|---|
| Portal selection screen | Shows both Coreweave and Vendor cards | Shows **only** the Coreweave card |
| Vendor card | Visible and clickable | **Hidden** |
| Coreweave card | Standard size, side-by-side layout | Full-width, single-column |
| Subtitle text | "Select your portal to get started" | "Sign in to scan and receive shipments" |

### 3.2 Implementation in `Login.jsx`

- Import `useMobileDetect` from `../utils/device`.
- Call `const isMobile = useMobileDetect()` at the top of the component.
- In the portal selection view:
  - If `isMobile`, do not render the Vendor card.
  - Change the subtitle to reflect the scanning purpose.
  - Apply `login-cards--mobile` class for full-width single-column layout.
- If someone bypasses detection (e.g., spoofed viewport), the backend/auth layer should still enforce role-based access — but for this client-side app, the UI gate is sufficient.

### 3.3 Deep Link Protection

If a mobile user manually navigates to a vendor route (e.g., `/vendor-portal`), the existing `ProtectedRoutes` in `App.jsx` already redirects non-vendor users. No additional route guarding needed.

## 4. Mobile Receiving Flow

After Coreweave login on a mobile device, the user lands on a **scan-first** receiving experience optimized for one-handed phone use.

### 4.1 Mobile Routing

In `App.jsx`, when `isMobile` is true and `user.role === 'coreweave'`:

| Route | Component | Purpose |
|---|---|---|
| `/` (index) | `MobileScanReceive` | Camera-based QR scanner + receiving checklist |
| `/receiving` | `MobileScanReceive` | Same (alias) |
| `*` | Redirect to `/` | All other routes redirect to scan |

The full desktop navigation (Dashboard, POs, Reconciliation, Data Management) is **not shown** on mobile. The sidebar collapses to a minimal header with the app name and a logout button.

### 4.2 `MobileScanReceive` Component

New file: `src/pages/mobile/MobileScanReceive.jsx`

#### Screen 1 — Scanner

- Full-screen camera viewfinder using the device's rear camera.
- Uses the browser `getUserMedia` API + a QR decoding library (e.g., `jsQR` or `html5-qrcode`).
- Overlay with crosshair and instruction text: "Point camera at pallet QR code."
- On successful scan, decode the JSON payload (Pallet ID, PO Number, SKU, MPN, quantity, serial numbers).
- Transition to Screen 2.
- Also include a "Manual entry" link that lets the user type/paste a PO number if no QR is available.

#### Screen 2 — Receiving Checklist

Once the manifest is loaded (from QR scan or manual PO lookup):

- **Header:** PO Number, Vendor, Tracking Number, Pallet ID (if scanned).
- **Line items as a scrollable checklist,** one card per item:
  - SKU + MPN (read-only)
  - Expected quantity (from manifest)
  - **Received quantity** — stepper control (+/−) defaulting to expected
  - **Serial verification** — each expected serial shown as a tappable chip:
    - Tap to mark as received (turns green)
    - Long-press to flag as missing/damaged (turns red)
    - Option to scan individual serial barcodes (Code 128) with the camera for automated matching
  - **Condition** — quick toggle: Good / Damaged / Wrong Item
  - **Notes** — expandable text field per item

#### Screen 3 — Confirmation

- Summary of what was confirmed: items received, serials verified, discrepancies flagged.
- Discrepancies highlighted in red with specific detail (e.g., "2 of 10 GPU units missing serial").
- **Sign & Submit** button:
  - Records the user's identity (from auth session) + timestamp.
  - Dispatches `RECORD_RECEIVING` to `AppContext`.
  - SOX compliance: the submission is the digital equivalent of a signed, dated packing slip.
- **"Scan Next Pallet"** button to return to Screen 1 for the next pallet in the same delivery.

### 4.3 Offline Considerations

Warehouse environments can have spotty connectivity.

- The app already uses localStorage for all state. Scans and checklist confirmations are saved locally immediately.
- If the app is offline at submission time, queue the receiving record and sync when connectivity returns.
- For the Capacitor native app (Section 6), this is handled more robustly via Capacitor's offline storage and background sync plugins.

## 5. Mobile Layout & CSS

### 5.1 New CSS Classes

| Class | Purpose |
|---|---|
| `.mobile-layout` | Replaces `.app-layout`; no sidebar, top bar only |
| `.mobile-header` | Compact top bar: app name, user badge, logout |
| `.mobile-scanner` | Full-viewport camera view with overlay |
| `.mobile-checklist` | Scrollable card list optimized for touch |
| `.mobile-checklist-item` | Single item card with tap targets ≥ 44px |
| `.mobile-serial-chip` | Tappable serial badge (pending / received / missing states) |
| `.mobile-stepper` | Quantity +/− control with large touch targets |
| `.login-cards--mobile` | Single-column full-width login card layout |

### 5.2 Rules

- All touch targets must be at least 44×44px (Apple HIG / Material guidelines).
- Font sizes on mobile receiving screens should be at least 16px to avoid iOS auto-zoom on input focus.
- Use `position: sticky` for the header so it remains visible while scrolling the checklist.
- Camera viewfinder must fill the viewport width and at least 60% of the height.

## 6. Option B — Capacitor Native App

### 6.1 What is Capacitor?

Capacitor (by Ionic) wraps the existing React web app in a native iOS/Android shell. The same codebase runs inside a WebView with access to native device APIs (camera, filesystem, push notifications, biometrics). No separate codebase needed.

### 6.2 Why Capacitor?

| Concern | Capacitor Advantage |
|---|---|
| App Store distribution | Installable from App Store / Google Play; appears as a "real app" |
| Camera access | Native camera API via `@capacitor/camera` — more reliable than browser `getUserMedia` |
| Barcode scanning | `@capacitor-community/barcode-scanner` — uses native OS barcode engine, faster and more accurate |
| Offline support | `@capacitor/filesystem` + `@capacitor/preferences` for robust local storage beyond browser limits |
| Push notifications | `@capacitor/push-notifications` — alert warehouse staff when a new shipment is incoming |
| Biometric auth | `@capacitor-community/biometric-auth` — Face ID / fingerprint for quick re-login |
| Auto-update | Capacitor supports live updates (via Appflow or custom) so the app updates without App Store review |

### 6.4 Code Changes for Capacitor

#### 6.4.1 Platform Detection

The `isMobileDevice()` utility in `src/utils/device.js` already checks `window.Capacitor?.isNativePlatform()`. When running inside the Capacitor shell, this returns `true` immediately — no viewport or UA sniffing needed.

#### 6.4.2 Scanner — Native vs. Browser

Create a scanner abstraction `src/utils/scanner.js`:

```
export async function scanQRCode() {
  if (window.Capacitor?.isNativePlatform()) {
    // Use @capacitor-community/barcode-scanner native engine
    return await nativeScan();
  }
  // Fall back to browser getUserMedia + jsQR
  return await browserScan();
}
```

The `MobileScanReceive` component calls `scanQRCode()` without caring which implementation runs.

#### 6.4.3 Storage — Capacitor Preferences

For the native app, supplement localStorage with Capacitor's `Preferences` API for larger, more reliable storage:

```
import { Preferences } from '@capacitor/preferences';

export async function saveReceivingQueue(data) {
  await Preferences.set({ key: 'receiving_queue', value: JSON.stringify(data) });
}
```

#### 6.4.4 Biometric Quick Login

On the native app, after initial email/password login, offer to enable Face ID or fingerprint for subsequent sessions. The biometric check unlocks the stored session token from secure storage — the user doesn't re-enter credentials.

### 6.6 When to Choose Browser vs. Native App

| Scenario | Recommendation |
|---|---|
| Quick rollout, minimal infrastructure | **Browser** — works today, no App Store approval needed |
| Reliable camera scanning in all warehouse conditions | **Capacitor native** — native barcode engine handles low light, distance, damaged codes better |
| Offline-first requirement | **Capacitor native** — robust offline queue with background sync |
| Push notification for incoming shipments | **Capacitor native** — requires native push capability |
| Biometric login for shared warehouse devices | **Capacitor native** — Face ID / fingerprint |
| Both simultaneously | Supported — same codebase, browser version and native app coexist |
