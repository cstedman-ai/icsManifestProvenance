# 20260325 — Server Storage — SQLite Gallery for Packing Slip Photos

> **Purpose:** Define how to introduce a lightweight TypeScript backend using SQLite3 so that annotated packing slip photos are persisted server-side on capture and viewable in a "Gallery" section from any Coreweave login session.
>
> **Related docs:**
> - `docs/methodologies/20260324.MobileReceiving.ScanAndVerify.md`
> - `docs/from.ICS/20260324.GuidingPrinciples.ReceivingAndNetsuiteIntegration.md`
> - `docs/problemSolutions/20260226.ItemProvenance.ReconciliationOfProcuredItems.md`

## 1. Rationale

Today the app is entirely client-side — state lives in `localStorage` and annotated packing slip PNGs download only to the user's device. This means:

- Photos are siloed on whichever phone captured them.
- A Coreweave desktop user cannot review what was scanned at the dock.
- There is no audit trail linking a photo to a PO, receiving event, or user across sessions.

**Goal:** Every photo captured on a mobile device is automatically uploaded to a server running alongside the Vite frontend. A SQLite database stores metadata (who, when, which PO, file path) and the images are stored on disk. Any Coreweave user — mobile or desktop — can browse a "Gallery" of all captured packing slips.

### Why SQLite3

- **Zero infrastructure** — a single file on disk, no Postgres/MySQL server to manage.
- **TypeScript-native** via `better-sqlite3` (synchronous, fast, typed).
- **Sufficient scale** — packing slip galleries are low-write, read-moderate workloads.
- **Portable** — the entire database can be copied, backed up, or migrated trivially.

## 4. Database Schema

### 4.1 `photos` Table

```sql
CREATE TABLE IF NOT EXISTS photos (
  id            TEXT PRIMARY KEY,          -- UUID
  filename      TEXT NOT NULL,             -- on-disk filename
  original_name TEXT NOT NULL,             -- original filename from device
  mime_type     TEXT NOT NULL DEFAULT 'image/png',
  file_size     INTEGER NOT NULL,          -- bytes
  width         INTEGER,                   -- pixel width (optional)
  height        INTEGER,                   -- pixel height (optional)

  -- Provenance
  captured_by   TEXT NOT NULL,             -- user email
  captured_at   TEXT NOT NULL,             -- ISO 8601
  date_stamp    TEXT NOT NULL,             -- YYYYMMDD:HHMM as shown on the image

  -- Linking
  po_id         TEXT,                      -- FK to purchase order (nullable if captured before PO selection)
  po_number     TEXT,                      -- denormalized for display
  vendor        TEXT,                      -- vendor name
  shipment_id   TEXT,                      -- FK to shipment (nullable)
  receiving_id  TEXT,                      -- FK to receiving record (nullable)

  -- Annotation metadata
  has_signature INTEGER NOT NULL DEFAULT 0, -- boolean
  has_annotations INTEGER NOT NULL DEFAULT 0,

  -- Audit
  created_at    TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_photos_captured_by ON photos(captured_by);
CREATE INDEX IF NOT EXISTS idx_photos_po_number ON photos(po_number);
CREATE INDEX IF NOT EXISTS idx_photos_captured_at ON photos(captured_at);
```

### 4.2 Why Not Store Blobs in SQLite?

Storing multi-megabyte PNGs as BLOBs in SQLite is possible but has downsides:

- **Database size** balloons quickly (one packing slip photo can be 2–8 MB).
- **Backups become slow** — the `.db` file grows unbounded.
- **Static serving** is simpler when files live on the filesystem.

**Decision:** Store images on the filesystem under `server/uploads/<YYYY>/<MM>/<uuid>.png`. SQLite stores only the metadata and relative file path.

### 6.1 Express Entry Point (`server/src/index.ts`)

```typescript

// API routes

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

### 6.2 Connection Singleton (`server/src/db/connection.ts`)

```typescript
import Database from 'better-sqlite3';
import path from 'path';

const DB_PATH = path.join(__dirname, '../../data/gallery.db');

let db: Database.Database | null = null;

export function getDb(): Database.Database {
  if (!db) {
    db = new Database(DB_PATH);
    db.pragma('journal_mode = WAL');   // Better concurrent read performance
    db.pragma('foreign_keys = ON');
  }
  return db;
}
```

### 6.3 File Upload with Multer

```typescript
import multer from 'multer';
import path from 'path';
import { v4 as uuid } from 'uuid';

const storage = multer.diskStorage({
  destination: (_req, _file, cb) => {
    const now = new Date();
    const dir = path.join(
      'uploads',
      String(now.getFullYear()),
      String(now.getMonth() + 1).padStart(2, '0')
    );
    // Ensure directory exists
    fs.mkdirSync(dir, { recursive: true });
    cb(null, dir);
  },
  filename: (_req, _file, cb) => {
    cb(null, `${uuid()}.png`);
  },
});

export const upload = multer({
  storage,
  limits: { fileSize: 15 * 1024 * 1024 }, // 15 MB max
  fileFilter: (_req, file, cb) => {
    if (file.mimetype.startsWith('image/')) {
      cb(null, true);
    } else {
      cb(new Error('Only image files are accepted'));
    }
  },
});
```

### 6.4 Thumbnail Generation

On upload, generate a smaller thumbnail for the gallery grid:

```typescript
import { createCanvas, loadImage } from 'canvas';

async function generateThumbnail(
  sourcePath: string,
  destPath: string,
  maxDim: number = 400
): Promise<void> {
  const img = await loadImage(sourcePath);
  const scale = Math.min(maxDim / img.width, maxDim / img.height, 1);
  const w = Math.round(img.width * scale);
  const h = Math.round(img.height * scale);

  const canvas = createCanvas(w, h);
  const ctx = canvas.getContext('2d');
  ctx.drawImage(img, 0, 0, w, h);

  const buffer = canvas.toBuffer('image/png');
  fs.writeFileSync(destPath, buffer);
}
```

**Alternative (no native dependency):** Use `sharp` instead of `canvas` for thumbnail generation — it's faster and doesn't require system-level Cairo:

```bash
npm install sharp
```

```typescript
import sharp from 'sharp';

async function generateThumbnail(src: string, dest: string): Promise<void> {
  await sharp(src).resize(400, 400, { fit: 'inside' }).png().toFile(dest);
}
```

## 7. Frontend Integration

### 7.1 Auto-Upload on Capture

When the user taps "Accept" in the annotation screen, the photo is:

1. **Downloaded to the device** (existing behavior — blob URL + `<a download>`)
2. **Uploaded to the server** in the background

```typescript
// src/utils/api.ts

const API_BASE = import.meta.env.VITE_API_URL || '/api';

export async function uploadPhoto(
  blob: Blob,
  metadata: {
    capturedBy: string;
    dateStamp: string;
    poId?: string;
    poNumber?: string;
    vendor?: string;
    shipmentId?: string;
    receivingId?: string;
  }
): Promise<{ id: string; fullUrl: string }> {
  const form = new FormData();
  form.append('photo', blob, `PackingSlip_${metadata.dateStamp}.png`);
  Object.entries(metadata).forEach(([key, val]) => {
    if (val) form.append(key, val);
  });

  const res = await fetch(`${API_BASE}/photos`, {
    method: 'POST',
    body: form,
  });

  if (!res.ok) throw new Error(`Upload failed: ${res.status}`);
  return res.json();
}
```

### 7.2 Upload Call in `handleAccept`

Inside the existing `handleAccept` function in `MobileScanReceive.jsx`, after the `toBlob` call that triggers the device download, fire the upload:

```javascript
out.toBlob(async (blob) => {
  if (!blob) return;

  // 1. Download to device (existing)
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `PackingSlip_${dateStamp}_${Date.now()}.png`;
  document.body.appendChild(a);
  a.click();
  document.body.removeChild(a);
  setTimeout(() => URL.revokeObjectURL(url), 5000);

  // 2. Upload to server (new)
  try {
    await uploadPhoto(blob, {
      capturedBy: userEmail,
      dateStamp,
      poId: selectedPO?.id,
      poNumber: selectedPO?.poNumber,
      vendor: selectedPO?.vendor,
    });
  } catch (err) {
    console.error('Photo upload failed — saved locally only', err);
    // Optionally queue for retry
  }
}, 'image/png');
```

### 7.3 Offline Resilience — Upload Queue

If the upload fails (offline, server unreachable), store the photo in IndexedDB and retry later:

```typescript
// src/utils/uploadQueue.ts

const DB_NAME = 'icsPhotoQueue';
const STORE_NAME = 'pending';

export async function queuePhoto(blob: Blob, metadata: object): Promise<void> {
  const db = await openDB();
  const tx = db.transaction(STORE_NAME, 'readwrite');
  tx.objectStore(STORE_NAME).add({
    id: crypto.randomUUID(),
    blob,
    metadata,
    createdAt: new Date().toISOString(),
  });
  await tx.done;
}

export async function flushQueue(): Promise<void> {
  const db = await openDB();
  const tx = db.transaction(STORE_NAME, 'readonly');
  const items = await tx.objectStore(STORE_NAME).getAll();

  for (const item of items) {
    try {
      await uploadPhoto(item.blob, item.metadata);
      const delTx = db.transaction(STORE_NAME, 'readwrite');
      delTx.objectStore(STORE_NAME).delete(item.id);
      await delTx.done;
    } catch {
      break; // Stop on first failure; retry next time
    }
  }
}
```

Call `flushQueue()` on app load and on `online` events.

## 11. Security Considerations

| Concern                | Mitigation                                                     |
|------------------------|----------------------------------------------------------------|
| Unauthenticated upload | Require a session token or API key on `POST /api/photos`       |
| File type spoofing     | Validate MIME type + magic bytes, not just extension            |
| Path traversal         | UUID filenames only — no user-supplied paths reach the filesystem |
| Large file DoS         | Multer `fileSize` limit (15 MB), rate limiting                 |
| Gallery access control | Only Coreweave role can view `/api/photos`; enforce in middleware |
| SQLite concurrency     | WAL mode + single-writer is fine for this workload             |

## 12. Migration Path from localStorage

The current app persists POs, shipments, and receivings in `localStorage`. When the backend is introduced:

1. **Phase 1 (this doc):** Only photos go through the server. PO/shipment/receiving state remains in localStorage.
2. **Phase 2:** Migrate all state to SQLite via additional tables and API endpoints, making the app fully server-backed.
3. **Phase 3:** Add user authentication (JWT or session-based) to replace the current role-only login.

This phased approach avoids a risky big-bang rewrite and delivers the gallery feature immediately.
